## Original Requirements

Hospital Triage Queue System

A system for emergency room intake that prioritizes patients based on clinical urgency. It ensures critical cases are surfaced to medical staff immediately and that the waiting queue always reflects current patient conditions in a high-pressure environment.

Core features:
- Register incoming patients with their name and presenting symptoms
- Assign each patient an urgency level from 1 (critical) to 5 (non-urgent) based on the symptoms
- Order the queue by urgency level first, then by arrival time within the same level
- Reorder the queue immediately whenever a new patient arrives or an existing patient is re-triaged to a different urgency level
- Let a physician take the patient at the head of the queue, removing them from the queue
- Show a live dashboard of the queue with each patient's urgency and waiting time

## Project Name

hospital_triage_queue

## Product Goals

- Maximize clinical safety by ensuring the most urgent patients are always prioritized
- Minimize physician cognitive load by providing a clear, real-time queue dashboard
- Enable rapid re-prioritization as patient conditions change without data loss

## User Stories

- As a triage nurse, I want to quickly register a new patient with their symptoms so that they are assigned an initial urgency and placed into the queue.
- As a physician, I want to see the queue sorted by urgency and arrival time so that I always treat the most critical patient next.
- As a physician, I want to re-triage a patient if their condition worsens so that the queue reflects their new priority immediately.
- As a charge nurse, I want to view a live dashboard showing each patient's urgency and waiting time so that I can monitor overall ED flow.
- As an emergency department manager, I want to ensure the system handles high-volume surges by automatically reordering the queue without manual intervention.

## Competitive Analysis

- Epic Emergency Department Triage: Integrated with EHR, but queue prioritization is static and requires manual re-sort
- Cerner FirstNet: Real-time tracking but limited symptom-based urgency assignment (mostly nurse-driven)
- Zoll Triage System: Well-suited for mass casualty events but not for day-to-day ED workflow
- Pulsara: Collaborative platform with messaging but lacks a visible priority queue
- Manual Whiteboard System: Common in many EDs but prone to errors and delayed updates
- TriageGO: Mobile app for triage but not designed for hospital queue management
- Ribbon Health: Referral coordination, not focused on ED intake queue

## Requirement Analysis

The system must support real-time queue ordering with a clear tie-breaking rule (urgency descending, then arrival time ascending). It needs a registration interface for nurses, a dashboard view for all clinical staff, and a safe dequeue operation for physicians. The urgency assignment should be algorithmic based on symptom mapping (e.g., ESI protocol) but allow manual override by triage nurse. The data model must handle concurrent updates (arrival, re-triage, removal) without corruption. Persistence should be lightweight (e.g., SQLite or in-memory) for proof-of-concept, but the architecture should allow scaling to a production DB. The dashboard should auto-refresh with sub-second latency.

## UI Design draft

The UI consists of a top bar with the system title and a 'New Patient' button. Below is a table showing the queue with columns: Position, Name, Urgency (color-coded badge – red for 1, orange for 2, yellow for 3, green for 4, blue for 5), Symptoms, Waiting Time (auto-updating). Each row has a 'Re-triage' dropdown and a 'Take' button for physicians. When 'New Patient' is clicked, a modal form appears with fields for Name, Symptoms, and an optional manual urgency override (default auto). The dashboard auto-refreshes every 500ms or uses WebSocket push. The layout is responsive, optimized for desktop but usable on tablets with larger touch targets.

## Anything UNCLEAR

The symptom-to-urgency mapping algorithm is not specified in detail. We will assume the ESI (Emergency Severity Index) standard as the base, with a configurable rule set. Additionally, the system's multi-user concurrency model (how to handle simultaneous re-triage and dequeue) needs clarification; we will implement an optimistic locking or queue-level mutex for the prototype.
