## Implementation approach

We will build the Hospital Triage Queue System as a web application using FastAPI (async Python web framework) for the backend, SQLite with SQLAlchemy for lightweight persistence, and a simple HTML/CSS/JavaScript frontend that polls the REST API every 500ms to simulate real-time updates. For concurrency, we will use an asyncio lock (asyncio.Lock) to protect queue mutations. The symptom-to-urgency mapping will be implemented as a configurable rule engine based on the ESI (Emergency Severity Index) standard, with a default mapping of common symptoms to urgency levels. The system will support patient registration, automatic urgency assignment, manual re-triage, physician dequeue, and a live dashboard showing colored urgency badges and auto-updating waiting times.

## Program call flow


sequenceDiagram
    participant Nurse as Nurse (Browser)
    participant UI as Frontend (static/script.js)
    participant API as FastAPI Backend (main.py)
    participant QM as QueueManager
    participant DB as Database (SQLite)
    participant RE as TriageRuleEngine

    Note over Nurse,UI: Registration
    Nurse->>UI: Fill form (name, symptoms, optional urgency)
    UI->>API: POST /register (name, symptoms, manual_urgency)
    API->>QM: add_patient(name, symptoms, manual_urgency)
    QM->>RE: assign_urgency(symptoms)
    RE-->>QM: urgency (int)
    QM->>DB: insert_patient(new_patient)
    DB-->>QM: success
    QM->>QM: _sort_queue()
    QM-->>API: return patient dict
    API-->>UI: JSON with patient data
    UI->>UI: update dashboard

    Note over Physician,UI: Dequeue
    Physician->>UI: Click "Take" on top patient
    UI->>API: POST /dequeue
    API->>QM: dequeue()
    QM->>QM: acquire lock
    QM->>DB: delete_patient(top_patient.id)
    DB-->>QM: success
    QM->>QM: remove from internal queue
    QM->>QM: release lock
    QM-->>API: return removed patient
    API-->>UI: JSON with removed patient
    UI->>UI: update dashboard

    Note over Nurse,UI: Re-triage
    Nurse/Physician->>UI: Select new urgency from dropdown
    UI->>API: POST /re_triage (patient_id, new_urgency)
    API->>QM: re_triage(patient_id, new_urgency)
    QM->>QM: acquire lock
    QM->>DB: update_patient(updated_patient)
    DB-->>QM: success
    QM->>QM: update urgency and triage_time
    QM->>QM: _sort_queue()
    QM->>QM: release lock
    QM-->>API: return updated patient dict
    API-->>UI: JSON
    UI->>UI: update dashboard

    Note over UI,API: Dashboard Polling
    UI->>API: GET /queue (every 500ms)
    API->>QM: get_queue()
    QM->>QM: compute waiting times (current_time - arrival_time)
    QM-->>API: list of patient dicts
    API-->>UI: JSON
    UI->>UI: render table with urgency badges and waiting times


## Anything UNCLEAR

The symptom-to-urgency mapping algorithm requires a clinical basis. We will implement a default rule set based on common ESI indicators (e.g., chest pain -> 1, severe abdominal pain -> 2, minor laceration -> 4) and make the rules configurable via a JSON file. Multi-user concurrency is handled via an asyncio lock on the QueueManager, but in a distributed deployment a database-level lock or optimistic concurrency would be needed. For this prototype, single-instance is assumed.
