## Implementation approach

We will build the platform using Flask as the web framework due to its simplicity and flexibility. Flask-SQLAlchemy will handle ORM with SQLite for development and PostgreSQL for production. Flask-Login manages user sessions, Flask-Mail handles email verification, and Werkzeug provides password hashing. Image uploads will be processed with Pillow and stored locally. WTForms will handle form validation. Pagination will use SQLAlchemy's limit/offset. The architecture follows a modular blueprint pattern with separate route files for auth, main, admin, and API endpoints. All data models are defined in a single models.py with clear relationships. The admin panel uses Flask-Admin for quick CRUD interfaces. Notifications are stored in a database table and fetched on page load. The system is designed to be easily deployable and maintainable.

## Anything UNCLEAR

The exact behavior of blocking is clarified as: when user A blocks user B, both users' posts are hidden from each other's feeds, they cannot send messages to each other, and any existing friend/follow relationships are removed. Notification preferences affect in-app notifications only (not email). Soft-delete means the user's data is marked with a deleted flag and hidden from normal views, but admins can still see it in the admin panel. The 'Saved Posts' list is a single default list; no custom collections are needed. The search uses simple LIKE queries on username, post content, and group name. The admin activity log records actions like ban, delete, verify with timestamp and admin ID.
